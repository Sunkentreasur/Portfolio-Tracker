Here is the architectural design review based on the provided PROJECT.md and ARCHITECTURE.md documents.I have evaluated the current architectural decisions against your stated goals of strict layer separation, long-term maintainability, offline-first operation, and financial data integrity.1. Dependency Inversion Violation in the Application LayerClassification: CriticalIssue: The dependency rules state that "The Application Layer may depend on the Domain and Infrastructure Layers".  Why it matters: If the Application layer directly depends on the Infrastructure layer (e.g., concrete SQLite databases, file system parsers, SQLAlchemy repositories), it becomes tightly coupled to those specific technologies. This makes it impossible to unit test the Application layer without mocking the entire database and file system, which violates the Dependency Inversion Principle (DIP) and compromises the goal to "keep components loosely coupled".  Trade-offs: Allowing direct dependency on Infrastructure speeds up initial development because you can import database models directly into your services. However, it severely degrades long-term maintainability and testability.Recommendation: Adopt strict Dependency Inversion. The Application layer should depend only on the Domain layer and abstract interfaces (e.g., TransactionRepositoryInterface). The Infrastructure layer should depend on the Application/Domain layers by implementing those interfaces.2. Missing Transactional Boundaries in the Import PipelineClassification: CriticalIssue: The Import Pipeline saves to the database at Stage 9 (Database Persistence), but continues with Stage 10 (Portfolio Recalculation) and Stage 11 (Analytics Refresh).  Why it matters: In financial software, data integrity is paramount. If the application crashes or throws an exception during recalculation (Stage 10) or analytics refresh (Stage 11), the raw data has already been persisted to the database (Stage 9). This could leave the application's internal state out of sync with the stored data.  Trade-offs: Fine-grained database commits use less memory but risk partial state updates on failure. Wrapping entire workflows in a single transaction uses more memory but guarantees atomicity (all-or-nothing execution).Recommendation: Implement a "Unit of Work" pattern in the Application layer. Ensure that stages 9, 10, and 11 are wrapped within a single transactional boundary so that any failure automatically rolls back the database insertion, preserving a consistent state.3. Analytics Recalculation Performance BottleneckClassification: CriticalIssue: The Analytics Pipeline recalculates holdings, cash flows, TWR, XIRR, CAGR, and benchmark comparisons dynamically. The success metrics state the app must remain responsive for "tens of thousands of transactions".  Why it matters: Calculating XIRR (which requires iterative root-finding algorithms) and running dynamic cash flows for tens of thousands of transactions across multiple brokerages is computationally expensive. If this pipeline runs synchronously on the main thread every time the UI requests a refresh, the PySide6 UI will freeze, directly violating the responsiveness goal.  Trade-offs: Dynamic calculation is mathematically pure and requires no complex caching architecture, but it scales poorly. Incremental calculation or caching adds state management complexity but ensures UI responsiveness.Recommendation: Introduce an Analytics Cache or materialized view strategy in the Infrastructure/Domain layers. Furthermore, mandate that the Application layer executes the Analytics Pipeline asynchronously (e.g., via background workers or threads) so the Presentation layer is never blocked.4. AI Assistant Offline Execution vs. Privacy AmbiguityClassification: CriticalIssue: The architecture includes an AI Assistant designed to analyze portfolio analytics and holdings. However, the core design principles mandate that the application is "Offline-first by default" and that "Security and privacy are fundamental requirements".  Why it matters: AI capabilities that process portfolio data typically rely on external cloud APIs (like OpenAI or Anthropic). Sending financial data to a cloud API directly contradicts the offline-first and private nature of the application. If the AI is meant to run completely offline (local LLM), it will impose massive CPU, RAM, and disk space requirements on the desktop environment.Trade-offs: Cloud-based AI is highly capable and easy to implement but compromises absolute privacy. Local AI protects privacy but vastly increases the complexity, size, and hardware requirements of the software.Recommendation: Explicitly define the AI Assistant's deployment model. If it uses a cloud API, design a strict "Data Sanitization/Anonymization" boundary in the Application layer and require explicit user consent before data transmission. If it is local, define the hardware constraints and model-loading infrastructure.5. UI Coupling in the Analytics PipelineClassification: RecommendedIssue: Stage 7 of the Analytics Pipeline dictates that the pipeline will "Generate charts and summaries" before returning results to the Presentation Layer. The technology stack includes PyQtGraph for charts and Matplotlib for reports.  Why it matters: The architecture strictly prohibits lower layers from depending on higher layers. If the Domain or Application layer generates concrete chart objects (like a PyQtGraph widget or a Matplotlib Figure), it couples your core business logic to specific presentation technologies.  Trade-offs: Generating charts in the backend is sometimes easier when heavily utilizing Pandas, but it ruins the separation of concerns and makes swapping UI frameworks impossible.Recommendation: Modify the Analytics Pipeline so that Stage 7 only generates raw, framework-agnostic statistical data and coordinate arrays (e.g., Pandas DataFrames, Python dataclasses). The Presentation Layer must be solely responsible for taking this raw data and rendering the actual PyQtGraph or Matplotlib charts.6. Write-Boundary Ambiguity in Plugin ArchitectureClassification: RecommendedIssue: The Plugin Architecture states plugins "must not directly modify the application's internal state" and must interact through public APIs. However, plugins are explicitly expected to provide "Brokerage statement importers".  Why it matters: A statement importer fundamentally must alter the application's state by inserting new transactions. If the boundary is too strict, importer plugins cannot function. If it is too loose, poorly written plugins could corrupt the database, violating the "Accuracy over convenience" principle.  Trade-offs: A restrictive API limits what open-source contributors can build. A permissive API risks database corruption and unexplainable calculations.Recommendation: Design a formal Command Bus or Data Submission API for plugins. Importer plugins should only be allowed to submit standardized TransactionProposal data transfer objects (DTOs) to the Application layer. The Domain layer will then validate these proposals and handle the actual database state mutation, ensuring plugins never write to SQLite directly.7. File Validation and Duplicate Detection ContextClassification: OptionalIssue: The Import Pipeline includes "Duplicate Detection" at Stage 7.  Why it matters: Detecting duplicate transactions often requires querying the existing database to see if a transaction ID or date/amount pair already exists. The architecture does not specify how the pipeline accesses this historical context without tightly coupling the parser to the database.Trade-offs: Injecting database access directly into the pipeline makes it stateful. Passing all existing database records into the pipeline uses excessive memory.Recommendation: Utilize the Specification pattern or inject a read-only TransactionExistsQuery interface into the Import Pipeline at Stage 7. This allows the pipeline to check for duplicates efficiently while remaining completely decoupled from SQLAlchemy.
